## 11:30 修复分享名片头像不可见的 P0 缺陷

### 根因分析
云存储权限设为「仅创建者可读写」后，被分享者无法直接访问 `cloud://` 格式的头像 URL。代码中存在 3 个 Bug 导致头像不渲染：

| 优先级 | Bug | 位置 | 描述 |
|--------|-----|------|------|
| P0 | `onShareAppMessage` 返回 `cloud://` 作为 `imageUrl` | preview/index.js:105 | WeChat 分享 API 无法识别 cloud:// 协议，导致转发卡片头像为空白/破裂 |
| P0 | `onShareTimeline` 同样问题 | preview/index.js:116 | 朋友圈分享也有相同缺陷 |
| P1 | `onAvatarError` 可能覆盖已解析的 HTTPS URL | preview/index.js:592 | 异步竞争：旧 cloud:// 图片加载失败触发 error → 默认头像覆盖了 resolveCloudUrls 刚解析的正确 HTTPS URL |
| P2 | Canvas 生成串行启动 | preview/index.js:200,210 | _generateShareCard 在 _resolveCardAvatar 内部触发，串行延迟导致用户快速点击分享时 Canvas 尚未完成 |

### 修复方案（仅修改 `pages/preview/index.js`）

**P0**: 分享回调中严格过滤 cloud:// 格式
- `onShareAppMessage`/`onShareTimeline`: imageUrl 优先级改为 `shareImagePath > HTTPS头像 > ''(空，微信自动生成默认卡片)`
- cloud:// 格式绝不传递给 WeChat API

**P1**: onAvatarError 添加防护
- 检查当前 avatar 是否已是 HTTPS URL（已被 resolveCloudUrls 解析）
- 若已是 HTTPS → 忽略该 error 回调（旧 cloud:// 请求失败的预期行为）
- 否则 → 正常降级为默认头像

**P2**: Canvas 生成并行化
- `_generateShareCard()` 从 `_resolveCardAvatar()` 内部移出
- 改为在 `loadCard().then()` 中与 `_resolveCardAvatar` 并行调用
- shareCard.js 内部独立处理 cloud:// → HTTPS，无需等待页面端解析

### 头像完整流转（修复后）

```
卡片创建者 A:
  上传头像 → wx.cloud.uploadFile → cloud://xxx → 存入 cards 集合

卡片创建者 A 自己查看:
  loadCard → card.avatar=cloud://xxx → <image> 自动解析(同openid有权限) → ✅

被分享者 B 打开分享链接:
  loadCard → card.avatar=cloud://xxx → <image> 尝试解析 → ❌ 权限拒绝
  → onAvatarError → 检测到仍是 cloud:// → 设置默认头像
  → _resolveCardAvatar 异步完成 → resolveCloudUrls(admin) → HTTPS URL
  → setData → <image> 重新加载 HTTPS → ✅ 头像正常显示！

卡片创建者 A 分享时:
  用户点分享 → onShareAppMessage
  → shareImagePath (Canvas已并行生成) → wxfile:// 本地文件 → ✅
  或: HTTPS头像 (已解析) → ✅
  或: 空 → 微信默认灰色卡片 → 可接受降级
  ❌ cloud:// 永远不会再传递给 WeChat API
```

## 12:00 裁切页画布拖拽边界扩大（方案一）

### 根因
`movable-area` = `screenWidth × screenWidth`（375×375），远超 movable-view 尺寸（420×281），
WeChat 内置边界算法 `[areaSize - viewSize, 0]` 导致拖拽范围仅 45px（横向），
用户无法将照片大幅拖动来调整裁切区域。

### 修改（2 文件，7 行）

| 文件 | 位置 | 改动 |
|------|------|------|
| `pages/crop/index.wxss:60-72` | `.crop-movable-area` | `top/left: 0→-100%`, `width/height: 100%→300%` |
| `pages/crop/index.js:62-63` | `onImageLoad` | `areaW/H: screenWidth→screenWidth*3` |
| `pages/crop/index.js:157-158` | `onConfirm` | `maskLeft/Top += screenWidth` (补偿 movable-area 偏移) |

### 效果验证
- **拖拽范围**: X `[-45,0]→[705,0]`, Y `[94,0]→[844,0]`（扩大 ~15×）
- **初始居中**: 数学验证等价（centerX 从 (-22.5,47) 变为 (352.5,422)，maskLeft 同步补偿）
- **裁切结果**: srcX/srcY 公式不变，结果完全一致
- **遮罩覆盖**: overflow:hidden + box-shadow 自动遮蔽溢出，视觉零变化

### 影响分析
- **零外部依赖**: crop 页仅被 `edit/index.js` 的 `onCropResult(tempFilePath)` 消费，接口不变
- **无回归**: `_doSquareCrop` / `_returnResult` / `onScale` / `onMove` / WXML 完全未修改
- **性能**: WXSS 纯布局变更，零额外渲染开销

| 位置 | 缺陷 | 严重度 |

   - 新增 `_resolveViaTempFileURL()` 辅助函数（212-231 行）

3. **`_loadAssets` 增强日志**（`shareCard.js:141-173`）
   - 无头像 / URL 解析失败 / 图片加载失败 / 成功 四个分支均有日志

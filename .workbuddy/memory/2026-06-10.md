## 15:44 修复首页空白 — babel 转译 + WXSS 兼容性

### 现象
- 首页空白，控制台报 `SystemError (appServiceSDKScriptError) timeout`
- 同时报 `ENOENT: babel_runtime_define_property.js not found`（DevTools 内部文件缺失）

1. crop 页面 JS 使用了大量 ES6+ 语法（`const`/`let`、箭头函数、解构、模板字符串），触发微信开发者工具的 babel 转译器需要 `define_property` 辅助文件，但该文件在 DevTools 的 asar 包中缺失
2. crop WXSS 中 `gap: 80rpx` 和 `calc(48rpx + env(safe-area-inset-bottom))` 混用 rpx + env() 在旧版基础库解析失败

- **JS**：全部改为 `var` + `function` 语法（ES5 风格），零箭头函数/解构/模板字符串/const/let
- **JS**：`wx.canvasToTempFilePath({...}, this)` → `wx.canvasToTempFilePath({canvas: canvas, ...})`（Canvas 2D 不需要 this 上下文）
- **JS**：`Math.max(0.3, cropSize / Math.max(imgW, imgH) * 0.5)` 拆分为清晰的 if 判断
- **WXSS**：移除 `gap` 属性 → 用 `.crop-btn + .crop-btn { margin-left: 80rpx; }` 替代
- **WXSS**：移除 `calc(48rpx + env(...))` → 固定 `padding: 24rpx 48rpx 72rpx`
- **WXSS**：`.loading-spinner` → `.crop-loading-spinner`，`.loading-text` → `.crop-loading-text`，`@keyframes spin` → `@keyframes cropSpin`（避免与其他页面全局样式冲突）
- **WXML**：同步更新 class 名

### 关键教训
- 微信开发者工具的 babel 转译器不稳定，避免在一个文件中混用过多 ES6+ 高级语法
- 小程序中所有 WXSS 是全局的，class 名冲突会导致跨页面样式污染
- `@keyframes` 名也是全局的，必须加页面前缀

## 15:58 裁切页「图片加载失败」第二轮修复 — 放弃预载图，改用可见图 + 加载遮罩

### 根因
上轮修复用 `left: -9999rpx` 隐藏预载图来获取尺寸，但微信小程序渲染引擎对屏幕外元素优化激进，
不渲染 → `bindload` 永远不触发 → 用户看到 "图片加载失败"。

### 新方案：裁切容器始终渲染 + 加载遮罩
**核心思路**：图片必须在可见的 movable-view 中渲染，`bindload` 才能 100% 触发。

- **WXML**：去掉隐藏预载图；`.crop-container` 不再 `wx:if="{{!isLoading}}"`（始终渲染）；
  在容器内放入 `.crop-loading-overlay`（`wx:if="{{isLoading}}"` 时显示）
- **JS**：`onLoad` 设置 `imgDisplayW=screenWidth`（初始可见尺寸），
  图片在 movable-view 中自然加载 → `onCropImageLoad` 获取 `e.detail.width/height`
  → 更新真正的 `imgDisplayW/imgDisplayH` + `isLoading: false`
- **WXSS**：`.crop-loading-overlay` — `position: absolute` 覆盖整个容器，`z-index: 10`，黑色半透明

### 关键教训
- 小程序 `<image>` 的 `bindload` 只在元素被实际渲染到可视区域时才触发
- `left: -9999rpx` / `opacity: 0` / `display: none` 都会导致引擎跳过渲染
- 获取图片尺寸的可靠方法只有：让图片在可见区域加载（哪怕被遮罩盖住）

## 16:12 修复裁切页图片缩小后自动放大（回弹 bug）

### 现象
用户手动缩小图片后，图片会自己逐步放大回去（回弹）

### 根因
`onScale` / `onMove` 中调用了 `setData({ currentScale, moveX, moveY })`，导致 movable-view
变成「受控组件」。每次手势触发 `bindscale` → `setData` 更新 `scale-value` → 组件重渲染 →
框架重新解释缩放值 → 再次触发 `bindscale` → 形成正反馈环路。

### 修复
初始渲染后不再通过 `setData` 控制 movable-view（转为非受控组件）：
- `onScale` / `onMove`：仅保存到 `_realX` / `_realY` / `_realScale`，不调用 `setData`
- `onConfirm`：用 `this._realScale` 替代 `data.currentScale` 获取当前缩放值
- `onImageLoad`：保留一次性 `setData({ currentScale, moveX, moveY })` 做初始定位

### 关键教训
- `movable-view` 的 `x`/`y`/`scale-value` 属性只能在初始化时设一次
- 手势交互期间持续 `setData` 更新这些值 = 受控组件反馈环路 → 回弹/stutter
- 正确姿势：初始 `setData` → 手势期间只记录到实例变量 → 确认时读取实例变量

## 17:49 修复裁切结果偏移（头像向左侧偏移）

### 现象
按遮罩裁切后的头像与遮罩下显示的图像不一致，结果向左侧偏移

### 根因
`onMove`（绑定 `bindchange`）中写了 `this._realScale = detail.scale`，
但 `bindchange` 事件的 `detail` 对象**不含 `scale` 字段**（只有 `{x, y, source}`）。
因此每次拖拽结束后 `_realScale` 被覆盖为 `undefined`。
`onConfirm` 回退到 `data.currentScale`（初始缩放值，非用户当前缩放值），
导致坐标换算用的缩放比错误 → 裁切窗口在源图上定位偏移 → 结果图像内容左偏。

### 修复
- `onMove`：删除 `this._realScale = detail.scale` 这一行
- `_realScale` 只在 `onScale`（bindscale，含 scale 字段）中更新
- 拖拽不改变缩放 → `_realScale` 保持上一次 bindscale 的正确值

### 关键教训
- WeChat `movable-view` 的 `bindchange` ≠ `bindscale`：
  - `bindscale`: detail = {x, y, scale}  ← 有 scale
  - `bindchange`: detail = {x, y, source} ← **无 scale**
- 混用两个事件时，不要假设它们返回相同的字段
